### BLOCKER 1 — PRACTICE-01: Practice timing never receives AMB passings

**Affected requirement:** PRACTICE-01

**Break location:** `ForwarderGrpcService.java` — `streamPassings.onNext()` block

```java
raceRepository.findFirstByStatus(RaceStatus.RUNNING).ifPresentOrElse(
    race -> eventPublisher.publishEvent(
        new LapPassingEvent(race.getId(), msg.getTransponderNumber(), msg.getRtcTimeMicros())),
    () -> log.debug("No running race — discarding passing for transponder {}",
        msg.getTransponderNumber())   // ← DISCARDS during practice sessions
);
```

`PracticeTimingService.onLapPassing()` listens for `LapPassingEvent`, but the event is never published when no race is RUNNING. During a practice session, every decoder passing is silently dropped. The entire chain (`PracticeTimingService` → `PracticeTimingHub` → `/topic/practice/{id}/timing`) receives nothing.

**Fix:** When `raceRepository.findFirstByStatus(RUNNING)` is empty, check `practiceSessionRepository.findRunningSession()` and publish with a sentinel `raceId` (e.g. `0L`) that `PracticeTimingService` filters on, or introduce a dedicated `PracticePassingEvent` type.

### BLOCKER 2 — AUDIO-06: Running-order scheduler never fires

**Affected requirement:** AUDIO-06

**Break location:** `RcTimingApplication.java` — `@EnableScheduling` absent

```java
@SpringBootApplication
@EnableAsync   // present
// @EnableScheduling  ← MISSING
public class RcTimingApplication { ... }
```

`RunningOrderAnnouncementService.checkAndAnnounce()` carries `@Scheduled(fixedRate = 10_000)`. Without `@EnableScheduling`, Spring's task scheduler is never started and the annotation is silently ignored. The 2-minute / 5-minute running-order interval announcements never fire for any race.

**Fix:** Add `@EnableScheduling` to `RcTimingApplication` (or to `AsyncConfig.java`).

### WARNING 1 — CTRL-06: Orphaned legacy transponder-link endpoint

Two endpoints exist for transponder linking; only one credits laps retroactively:

| Path | Effect |
|------|--------|
| `POST /api/v1/race-control/race/{raceId}/unknown-transponder-link` (Phase 4 legacy) | Writes to `unknown_transponder_links` (V18 schema). `LapTimingService` never reads this table. **No timing effect.** |
| `POST /api/v1/race-control/races/{raceId}/transponders/link` (Phase 5 working) | Calls `lapTimingService.linkTransponder()` → `retroactiveLinkTransponder()` → STOMP broadcast. **This is the correct path.** |

**Risk:** If the cockpit frontend calls the older URL (singular `race/`), the race director sees no error but zero laps are credited. The legacy endpoint should be removed or documented as no-op.

---

### WARNING 2 — EVENT-05: Undocumented `ENTRIES_CLOSED` state

The requirements specify `DRAFT → PUBLISHED → OPEN → IN_PROGRESS → COMPLETED`. The implementation adds `ENTRIES_CLOSED` between `OPEN` and `IN_PROGRESS`:

`DRAFT → PUBLISHED → OPEN → ENTRIES_CLOSED → IN_PROGRESS → COMPLETED`

`EntryService.submitEntry()` only allows entry in `OPEN` or `PUBLISHED`. Any client driving `OPEN → IN_PROGRESS` directly receives a 409 conflict. Low operational risk (the state is functional), but the requirement contract is violated and no tests validate this transition exists.

---

### WARNING 3 — FORMAT-09: Qualifying seeding is manual, not event-driven

`QualifyingStandingsService.recalculateStandings()` correctly sorts entries by FTQ (laps DESC, best lap ASC). However, there is no code that extracts qualifying results from persisted `ResultSnapshot` objects and pipes them into `recalculateStandings()` after qualifiers complete. The admin must manually `POST /api/v1/admin/events/{id}/seed-finals` with pre-computed per-entry statistics. The algorithm is correct; the automation trigger is missing.

---

### WARNING 4 — AUDIO-09: Clip cache not cleared on race finish

`AudioPreGenerationService.clearClips(Long raceId)` is defined but called from nowhere. The `ConcurrentHashMap` clip cache accumulates indefinitely across all races in a day. On a full-day event (50+ races), this is a memory leak.

**Fix:** Call `clearClips(raceId)` from `onRaceStatusChanged()` when `newStatus == RaceStatus.FINISHED`, or wire it to a `ResultSnapshotService` hook.
