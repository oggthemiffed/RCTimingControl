## Interaction Contract

### '?' Button in Layout Headers

- **Placement:** Top-right of each layout's header bar, inside the existing `ml-auto` flex group.
  - `RaceControlLayout`: add before the logout button in `<div className="ml-auto flex items-center gap-3">`.
  - `AdminPanelLayout` desktop: not applicable (sidebar layout â€” add to mobile top bar `<header>` after brand text).
  - `AdminPanelLayout` mobile header: add to the right of the `<span>` brand label.
  - Racer portal layout (if it has a header): same `ml-auto` pattern.
- **Appearance:** `<Button variant="ghost" size="icon-sm" aria-label="Open help">` containing a `<HelpCircle className="h-4 w-4" />` lucide icon.
- **Tooltip:** The button is icon-only (no visible label). Wrap it in a shadcn `<Tooltip>` with `<TooltipContent>Open help</TooltipContent>` so sighted users hovering over the button see a label. If `Tooltip` is unavailable in a given layout context, fall back to a native `title="Open help"` attribute on the `<Button>` element.
- **Behaviour:** Clicking opens the `Sheet` from `HelpProvider` context. If the current page has not called `useHelp()`, the button is hidden (or renders as `null`) â€” pages without help content do not show a broken drawer.
- **State:** Button disabled (visually suppressed with `opacity-50 cursor-default`) when no help content is registered for the current page.

### Help Drawer (Sheet)

- **Component:** `<Sheet>` with `side="right"`.
- **Width:** `w-96` (384px) on desktop, full-width on mobile (shadcn Sheet default).
- **Header:** `<SheetHeader>` with `<SheetTitle>` showing the page name (e.g. "Race Control Help"). `<SheetDescription>` with one sentence describing the page purpose.
- **Body:** Scrollable `overflow-y-auto` area containing the help article JSX component registered by the current page.
- **Footer:** Pinned at bottom of drawer â€” contains a link (rendered as `<Button variant="outline" size="sm" asChild><Link to="/print/...">`) to the relevant printable guide. Text: "Open Race Meeting Guide", "Open Racer Guide", or "Open Admin Guide" depending on context. Link opens in a new tab (`target="_blank"`).
- **Close:** Standard shadcn Sheet close button (top-right X). ESC key closes. Clicking outside closes.

### Help Article Format (in-app)

Each `frontend/src/help/XxxHelp.tsx` component renders:
1. Two to three sentences of prose explaining what the page does â€” `text-sm text-muted-foreground`.
2. A bulleted list (`<ul className="mt-3 space-y-1.5 text-sm">`) of 3â€“5 key actions available.
3. A "Common mistakes" callout (`<div className="mt-4 rounded-md bg-muted p-3 text-sm">`) with one to two pitfalls.

No external dependencies. No markdown parsers. Plain JSX with Tailwind/shadcn classes.

### Printable Guide Pages

- Route pattern: `/print/meeting-guide`, `/print/racer-guide`, `/print/admin-guide`.
- Routes are unprotected (same as existing `/print/race/:raceId`).
- Page structure matches `PrintResultsPage.tsx` exactly:
  - `useEffect` sets `document.title`.
  - Outer wrapper: `<div className="p-8 max-w-3xl mx-auto print:p-4">`.
  - No nav, no sidebar, no header chrome.
  - Print button: `<button onClick={() => window.print()} className="... print:hidden">Print / Save as PDF</button>`.
- Club name injected via `useClubProfile()` hook (pattern already exists) or a static placeholder if the hook is unavailable in print context.

## Constraints & Notes

1. **No new dependencies.** This phase adds zero npm packages. All components reuse existing shadcn/ui installed in prior phases.
2. **Icon choice.** Use `HelpCircle` from `lucide-react` for the '?' button. This is consistent with the existing lucide-react usage throughout the codebase (despite `components.json` declaring `remixicon`).
3. **HelpProvider placement.** Wraps inside `AuthProvider` and `QueryProvider` in `App.tsx` (so `useAuth()` is available to help content that may need it). Analogous to the `SetupGuard` wrapping pattern from Phase 8.
4. **Print routes are unprotected.** Consistent with existing `/print/race/:raceId` route. No `ProtectedRoute` wrapper.
5. **Content is written by the executor agent.** The executor reads the implemented pages before authoring help text. David reviews accuracy before phase sign-off (D-11 from CONTEXT.md).
6. **Racer portal layout.** If the racer portal does not have a named layout with a persistent header, the '?' button is added to the individual racer page headers using a shared `PageHeader` component pattern. The executor determines the best placement after reading racer layout code.
