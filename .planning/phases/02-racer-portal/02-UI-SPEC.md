## Spacing Scale

Declared values (multiples of 4, Tailwind v4 defaults apply):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, badge internal padding |
| sm | 8px | Compact row spacing, input internal gap |
| md | 16px | Default form field spacing (`space-y-4`), card padding |
| lg | 24px | Section padding, card header/content gap |
| xl | 32px | Layout column gaps, page section breaks |
| 2xl | 48px | Between major portal sections (e.g. profile header → form) |
| 3xl | 64px | Page top/bottom padding on desktop |

Exceptions:
- Mobile bottom nav bar: 56px height (14px icon + 4px gap + 12px label + 2×12px vertical padding) — thumb reach target. Implemented as `h-14` (56px).
- Touch targets on bottom nav items: minimum 44px tap height enforced by `py-3` per item.
- Car card tag grid cells: 12px horizontal padding (between md and sm) — use `px-3`.

Font: Figtree Variable, sans-serif. `--font-sans` and `--font-heading` both resolve to Figtree Variable (VERIFIED: index.css).

| Role | Size | Weight | Line Height | Tailwind Class | Usage |
|------|------|--------|-------------|----------------|-------|
| Body | 14px | 400 (regular) | 1.5 | `text-sm` | Form labels, card body text, table cells, nav item labels |
| Body prominent | 16px | 400 (regular) | 1.5 | `text-base` | Input values, primary paragraph content |

Weights used: 400 (regular) and 600 (semibold) only. No 700/bold.

Note: The auth layout already uses `text-2xl font-semibold` for the app name — maintain this precedent for all page display headings.

## Color

All color values expressed as CSS variable references (resolved by shadcn taupe/radix-luma theme in index.css). OKLCH values are already set; use semantic token names in code, not raw OKLCH values.

| Role | CSS Variable | Light Mode Approx | Dark Mode Approx | Usage |
|------|-------------|-------------------|------------------|-------|
| Dominant (60%) | `--background` | White (#fff) | Near-black warm | Page background, main content area, auth card outer area |
| Secondary (30%) | `--card` / `--muted` | White / Light taupe | Dark warm grey | Portal layout cards, section containers, bottom nav background, muted row backgrounds |
| Accent (10%) | `--primary` | Mid blue (oklch 0.5 0.134 242) | Darker blue | Reserved list below |
| Destructive | `--destructive` | Red-orange (oklch 0.577 0.245 27) | Lighter red | Archive car button, withdraw entry button — destructive actions only |

Accent (`--primary` / `text-primary`) reserved for:
- Active nav item indicator (top nav active link, bottom nav active icon + label)
- Primary CTA buttons (`Button` variant `default` — already uses `bg-primary`)
- Form field focus ring (`--ring` resolves to taupe ring, `--primary` used for active state)
- Entry status badge for CONFIRMED state
- Links within body text (`text-primary underline underline-offset-4` — established in auth screens)

Secondary semantic colors:
- `--muted-foreground`: Secondary descriptive text, card descriptions, empty state body copy
- `--secondary`: Bottom nav background on mobile, inactive nav pill backgrounds
- `--destructive`: Archive/withdraw confirmation buttons only — never used for styling warnings

Entry status badge colors:
- CONFIRMED: `bg-primary/10 text-primary` (accent tint)
- WITHDRAWN: `bg-muted text-muted-foreground` (neutral)
- OPEN (event status): `bg-primary/10 text-primary`
- CLOSED (event status): `bg-muted text-muted-foreground`
- UPCOMING (event status): `bg-secondary text-secondary-foreground`

## Layout Patterns

### Portal Shell (`RacerPortalLayout.tsx`)

The layout wraps all `/racer/*` routes. Source: D-06, Pattern 3 from RESEARCH.md.

```
Desktop (md and above):
  [Top nav: Profile | Cars | Transponders | Entries]    (border-b, px-4 h-14)
  [Page content via <Outlet />]                          (flex-1, container max-w-2xl)

Mobile (below md):
  [Page content via <Outlet />]                          (flex-1, pb-16 to clear bottom nav)
  [Bottom nav: fixed bottom-0, h-14, 4 items]           (bg-background border-t)
```

Bottom nav items: Profile (ri-user-line), Cars (ri-car-line), Transponders (ri-wifi-line), Entries (ri-file-list-line). Active item uses `text-primary`. Inactive items use `text-muted-foreground`.

Top nav active link: `text-primary border-b-2 border-primary` (underline indicator below text). Inactive: `text-muted-foreground hover:text-foreground`.

### Page content width
All portal pages: `container max-w-2xl mx-auto px-4 py-6` on desktop. Full width (no container) on mobile with `px-4`.

### Public event schedule (`/events`)
No auth shell. Standalone page. Centered layout: `container max-w-3xl mx-auto px-4 py-8`.

## Page-by-Page Interaction Contract

### `/racer/profile` — ProfilePage

**Layout:** Single-column form. Section: Personal Details (firstName, lastName, phoneNumber). Section: Emergency Contact (emergencyContactName, emergencyContactPhone). Section: Governing Body Memberships (list of membership rows + Add Membership form). Section: Ability Ratings (read-only table, no edit controls).

**Interactions:**
- Profile form: inline edit (fields editable in place, not behind a button). Save button appears fixed at bottom of form when dirty (React Hook Form `isDirty`). Save button label: "Save changes".
- Membership rows: add via inline form at bottom of list. Delete via icon button (ri-delete-bin-line) with Dialog confirmation.
- Ability ratings: read-only. No edit affordance on portal. Show as table: Class | Rating (0–100) | visual bar (Tailwind `w-[{rating}%] bg-primary/30`).

**States:**
- Loading: Skeleton rows (use `animate-pulse bg-muted rounded` divs) for each form section.
- Empty memberships: "No memberships on file. Add your first below."
- Empty ability ratings: "No ability ratings assigned yet."
- Save pending: Button shows `<Loader2 className="animate-spin" />` + "Saving…"
- Save success: Sonner toast "Profile updated"
- Save error: Sonner toast "Failed to save. Please try again."

### `/racer/cars` — CarsPage

**Layout:** Card list. Each car = one Card with car name as CardTitle, primary class as CardDescription, tag values as a compact `flex flex-wrap gap-2` row of plain text pairs (`Chassis: Team Associated` etc.). Add Car button at top-right of page heading.

**Interactions:**
- Edit car: Click "Edit" (Button variant=`ghost` size=`sm`) on card → opens Sheet from right side. Sheet contains car name input, notes textarea, tag value inputs (one per category). Sheet title: "Edit {car name}". Sheet save button: "Save car".
- Archive car: Button variant=`ghost` size=`sm` with destructive icon (ri-archive-line) → Dialog confirmation. Dialog title: "Archive {car name}?". Body: "Archived cars are hidden from entries but their race history is preserved." Confirm button: "Archive car" (destructive variant).
- Add car: Sheet opens from right side (same Sheet pattern). Sheet title: "Add a car". Save button: "Add car".

**States:**
- Loading: 3 skeleton cards.
- Empty: "You haven't added any cars yet." + "Add your first car" Button.
- Car save pending: Sheet save button shows spinner.
- Car save success: Sheet closes, list refreshes via `invalidateQueries(['racer', 'cars'])`, Sonner toast "Car saved".
- Archive success: Card disappears from list, Sonner toast "{car name} archived".

### `/racer/transponders` — TranspondersPage

**Layout:** Card list (same pattern as cars — consistent with D-07 intent). Each transponder = one Card. CardTitle: transponder label (or transponder number if no label). CardDescription: transponder number in monospace (`font-mono text-sm`). No inline edit — transponder number is immutable after creation. Add Transponder button at top-right.

**Interactions:**
- Add transponder: Small inline form above the card list (not a Sheet — simpler data). Fields: transponder number (required), label (optional). Submit button: "Add transponder".
- Delete transponder: Icon button (ri-delete-bin-line, variant=`ghost` size=`icon`) → Dialog confirmation. Dialog title: "Remove transponder {number}?". Body: "This transponder will be removed from your account. Existing race entries that used it are not affected." Confirm: "Remove transponder" (destructive).

**States:**
- Loading: 3 skeleton cards.
- Empty: "No transponders registered." + "Add your first transponder" (inline form auto-focused).
- Duplicate transponder error (409): Inline form error under number field: "This transponder number is already registered in the system. Contact a club official if you believe this is an error."
- Add success: Form clears, list refreshes, Sonner toast "Transponder added".

### `/racer/entries` — EntriesPage

**Layout:** Two sections. Section 1: "Open Events" — cards for events with status OPEN showing entry CTA. Section 2: "My Entries" — table (TanStack Table) showing past and current entries.

Table columns: Event | Date | Class | Transponder | Status | Actions.
Status column renders Badge component.
Actions column: "Withdraw" Button (variant=`ghost` size=`sm`, destructive color via `text-destructive`) shown only when entry is CONFIRMED and event is still OPEN.

Entry submission form: Opens as Dialog (not a page). Dialog title: "Enter {event name}". Fields: Racing Class (Select), Car (Select — active cars only), Transponder (Select). Submit button: "Submit entry".

**States:**
- Loading entries table: Skeleton rows (5 rows, 4 columns).
- Empty entries table: "You haven't entered any events yet. Enter an open event above to get started."
- No open events: "No events are currently open for entry." (plain text, no empty state illustration)
- Entry submission pending: Submit button shows spinner + "Submitting…"
- Entry submission success: Dialog closes, Sonner toast "Entry confirmed for {event name}", entries list refetches.
- Membership blocked (422): Dialog error banner: "Entry blocked — governing body membership required. Contact a club official if you need an override." No dismiss — racer must close dialog.
- Transponder conflict warning (200 + warning body): Dialog shows yellow `bg-amber-50 border border-amber-200` banner: "Note: this transponder is also used by another entry at this event. Check with the club if your classes run at the same time." Warning does not block submission — "Submit entry" button remains enabled.
- Withdraw pending: Button shows spinner.
- Withdraw success: Row status badge updates to WITHDRAWN, Sonner toast "Entry withdrawn".

### `/events` — EventSchedulePage (public, no auth)

**Layout:** Standalone page, no portal shell. Page heading: "Event Schedule". Subheading: club name (from club profile endpoint if available, otherwise omit). Table: Event | Date | Status | Action. No login-required state — page is always accessible. Link to login when entry link is clicked without a session.

Status column renders Badge: OPEN (accent), CLOSED (muted), UPCOMING (secondary/neutral).
Action column: "Enter event" Button (variant=`default`, size=`sm`) when OPEN. Clicking while not logged in redirects to `/login?next=/racer/entries`.
Empty table: "No upcoming events scheduled."

## Accessibility Contract

- All interactive elements (nav items, buttons, form fields) must have accessible labels. Use shadcn component ARIA patterns — do not add custom ARIA that conflicts with Radix.
- Bottom nav items: `aria-label="{page name}"` on each NavLink. Active item: `aria-current="page"`.
- Sheet: focus trap is handled by Radix SheetContent — do not override.
- Dialog: focus trap is handled by Radix DialogContent — do not override. Confirm button should NOT be auto-focused (prevent accidental destructive action on Enter key).
- Form validation errors: `FormMessage` from shadcn/ui renders with `role="alert"` via Radix — do not replace with custom error display.
- Transponder number field: `inputMode="numeric"` for mobile keyboard optimization.
- Entry status badges: include visually-hidden text when badge is the only status indicator (e.g. `<span className="sr-only">Status: </span>` before badge text).
- Color is never the sole differentiator for status — badge text label always accompanies color.

## Implementation Notes for Executor

1. **Lucide vs Remixicon:** The existing auth screens use `Loader2` from `lucide-react` (package already installed). The shadcn preset specifies Remixicon as the icon library. Use `ri-*` CSS classes for all new Phase 2 icons. Reserve `Loader2` from lucide-react only for spinner animations (existing pattern) until a Remixicon spinner equivalent is confirmed.

2. **`isLoading` vs `isPending`:** TanStack Query v5 (installed: 5.99.0) uses `isPending` not `isLoading` for the initial query loading state. Use `isPending` consistently. The auth screens use `useState` for mutation pending — racer portal hooks should use `useMutation.isPending` directly.

3. **Sheet side:** All edit sheets open from `side="right"`. Width: default shadcn Sheet width (fits mobile at full-width breakpoint). Do not override width.

4. **`RacerPlaceholderPage.tsx` wildcard route:** Replace the `path: '/racer/*'` wildcard entry in App.tsx with a `children:` array on `path: '/racer'`. The placeholder file can be deleted.

5. **Entry form Select values:** Event class ID and car ID are numbers. shadcn Select passes strings via `onValueChange`. Use `Number(value)` when building the submit payload. Validate with Zod `z.coerce.number()` for these fields.

6. **Sonner toast:** Already wired via `<Toaster />` in App.tsx and `toast` import from `sonner`. Follow the established pattern: `toast.error(message, { duration: 8000 })` for errors, `toast.success(message)` for success (default duration).

7. **Public event schedule auth redirect:** When an unauthenticated user clicks "Enter event", redirect to `/login?next=/racer/entries`. The `ProtectedRoute` component should read the `next` query param on successful login.
