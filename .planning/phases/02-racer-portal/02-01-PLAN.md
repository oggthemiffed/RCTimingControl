       ```kotlin
       jooq {
           version.set("3.19.11")
           configurations {
               create("main") {
                   generateSchemaSourceOnCompilation.set(true)
                   jooqConfiguration.apply {
                       logging = org.jooq.meta.jaxb.Logging.WARN
                       jdbc.apply {
                           driver = "org.postgresql.Driver"
                           url = System.getenv("JOOQ_JDBC_URL") ?: "jdbc:tc:postgresql:16-alpine:///jooq"
                           user = System.getenv("JOOQ_JDBC_USER") ?: "test"
                           password = System.getenv("JOOQ_JDBC_PASSWORD") ?: "test"
                       }
                       generator.apply {
                           name = "org.jooq.codegen.DefaultGenerator"
                           database.apply {
                               name = "org.jooq.meta.postgres.PostgresDatabase"
                               inputSchema = "public"
                               excludes = "flyway_schema_history"
                           }
                           generate.apply {
                               isRecords = true
                               isImmutablePojos = false
                               isFluentSetters = false
                           }
                           target.apply {
                               packageName = "dev.monkeypatch.rctiming.jooq.generated"
                               directory = "build/generated-sources/jooq"
                           }
                           strategy.name = "org.jooq.codegen.DefaultGeneratorStrategy"
                       }
                   }
               }
           }
       }
       ```
    4. Wire the codegen task to run Flyway against the Testcontainers DB first. The Testcontainers JDBC URL (`jdbc:tc:postgresql:16-alpine:///jooq`) auto-creates a container and applies classpath migrations. To ensure Flyway migrations run against it, add:
       ```kotlin
       tasks.named("generateJooq") {
           // Flyway runs automatically via Testcontainers JDBC + classpath:db/migration
           // (TC reads classpath:/db/migration by default when using `jdbc:tc:` URL)
           inputs.dir("src/main/resources/db/migration")
       }
       ```
       Testcontainers' `jdbc:tc:` URL automatically picks up `src/main/resources/db/migration/*.sql` when `TC_INITSCRIPT` or `TC_INITFUNCTION` is configured, OR the simpler route: add a `tmpfs` Postgres via docker compose. Use the Testcontainers approach — it is the stated project standard (AbstractIntegrationTest uses Testcontainers).

       Alternative (simpler, recommended): add explicit Flyway migration step before codegen:
       ```kotlin
       import org.flywaydb.gradle.task.FlywayMigrateTask

       // At plugin block:
       plugins {
           id("org.flywaydb.flyway") version "10.20.1"
           // existing plugins...
       }

       // Configure Flyway against the TC URL:
       flyway {
           url = System.getenv("JOOQ_JDBC_URL") ?: "jdbc:tc:postgresql:16-alpine:///jooq"
           user = System.getenv("JOOQ_JDBC_USER") ?: "test"
           password = System.getenv("JOOQ_JDBC_PASSWORD") ?: "test"
           locations = arrayOf("classpath:db/migration")
       }

       tasks.named("generateJooq") {
           dependsOn("flywayMigrate")
       }
       ```
       Use whichever of these two wiring approaches compiles without error. Prefer the explicit flyway plugin route for clarity.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Tampering | Flyway migrations | mitigate | All SQL files checked into git under `db/migration/`; Flyway checksum catches tampering on startup |
| T-02-01-02 | Information Disclosure | Generated jOOQ DSL | accept | Generated classes are build artifacts, not runtime secrets; contain only public schema metadata |
| T-02-01-03 | Elevation of Privilege | shadcn CLI fetches remote components | mitigate | Registry is ui.shadcn.com (official); components are reviewed manually after install before committing |
| T-02-01-04 | Tampering | `events` table `status` CHECK constraint bypass | mitigate | DB CHECK constraint in V12 rejects any status outside the allowed set; enum in Java will match |

No high-severity threats. Foundation-only plan — no runtime attack surface.
</threat_model>
