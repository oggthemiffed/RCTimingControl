**REST contract (follow /api/v1/ convention):**
```
GET /api/v1/race-control/race/{raceId}/pre-race-readiness
  Auth: Bearer JWT with role RACE_DIRECTOR or ADMIN
  200 OK → PreRaceReadinessDto
  403    → Insufficient role
  404    → Race not found
```

**Response shape:**
```java
public record PreRaceReadinessDto(
    long raceId,
    String raceLabel,          // e.g. "Qualifying 2 — Modified Touring — Heat 2"
    boolean firstRaceOfEvent,  // true when marshalDuty is intentionally empty
    List<GridCallSlotDto> gridCall,      // grid_position ASC, null positions last
    List<MarshalDutyRowDto> marshalDuty  // ordered by finishing position of previous race if known, else by driverName ASC
) {}
public record GridCallSlotDto(int gridPosition, long entryId, String driverName, String carNumber, String className) {}
public record MarshalDutyRowDto(long entryId, String driverName, String carNumber, long missedThisEvent) {}
```

`raceLabel` format (for both cockpit header and toast copy D-20):
- PRACTICE/QUALIFIER: `"{Practice|Qualifying} {roundNumber} — {className} — Heat {heatNumber}"`
- FINAL: `"{finalLetter} Final — {className}"` (e.g. `"A Final — Modified Touring"`)

**Previous-race resolution rule (CTRL-07, D-20):**
The marshal duty list is the driver set from the race immediately preceding the target race in the full event run order. Preceding = highest `(round.sequence_in_event, race.sequence_in_round)` tuple less than the target's. If no earlier race exists for the event, `firstRaceOfEvent = true` and `marshalDuty = []`.

**Absence count source (D-21):**
`SELECT COUNT(*) FROM marshal_absences WHERE entry_id = :entryId AND event_id = :eventId` — exposed via existing `MarshalAbsenceRepository.countByEntryIdAndEventId` (plan 02). For this read-side query, join `MARSHAL_ABSENCES` directly in the jOOQ SELECT and aggregate with `count().filterWhere(...)` or a correlated subquery per row.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → pre-race-readiness API | Untrusted authenticated client requests race metadata. Role-gated to RACE_DIRECTOR / ADMIN; racer tokens must be rejected. |
| jOOQ query → database | Read-only path. User-supplied `raceId` flows into bound parameters. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Elevation of Privilege | PreRaceReadinessController | mitigate | `@PreAuthorize("hasAnyRole('RACE_DIRECTOR','ADMIN')")` at method level; racer tokens receive 403 via Spring Security. Verified by IT `getPreRaceReadiness_racerRole_returns403`. |
| T-04-04-02 | Tampering / SQL Injection | PreRaceReadinessQuery | mitigate | All user input (`raceId`) is bound via jOOQ parameterised queries (`.eq(raceId)`) — no string concatenation. No dynamic table or column names. |
| T-04-04-03 | Information Disclosure | MarshalDutyRowDto | accept | Driver names and marshal-absence counts are already visible to race directors in the pre-race readiness view (the core feature). No additional PII beyond what the existing racer entry list already exposes. |
| T-04-04-04 | Denial of Service | PreRaceReadinessQuery | mitigate | Query is bounded to a single race + one preceding race + one event's driver count. Indexes on `race_entries.race_id` (V17) and `marshal_absences.event_id` (V18) keep cost O(participants_per_race). |
</threat_model>
