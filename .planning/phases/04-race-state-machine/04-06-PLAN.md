**Pitfall 1 — SecurityConfig:** Add `.requestMatchers("/ws/timing").permitAll()` in the existing authorizeHttpRequests chain (before `.anyRequest().authenticated()`). JWT is validated at STOMP CONNECT, not at HTTP upgrade.

**Pitfall 3 — concurrency:** All mutations of a given LiveRaceState go through `synchronized (state) { ... }` blocks inside LapTimingService. Position recalculation is O(n log n) over typically ≤40 entries; negligible.

**Pitfall 6 — unknown transponder dedupe:** LiveRaceState holds `Set<String> seenUnknownTransponders`; only the first passing per race per unknown transponder publishes a STOMP notification. This plan only implements the server-side state; the notification topic is reused (`/topic/race/{raceId}/state` with a discriminated message type, or a new `/topic/race/{raceId}/unknown-transponder` — choose the latter for clarity).

**CTRL-09 skip-to scope clarification:** In Phase 4 the active-race override is held in an in-memory field on `RaceControlService` (or directly on the controller as a `ConcurrentHashMap<Long, Long> eventIdToActiveRaceId`). This is process-local — if the JVM restarts or a second instance is added, clients re-derive the active race from the run-order list by finding the first non-FINISHED race. Cross-session persistence of the override is deferred beyond Phase 4. The endpoint validates that `targetRaceId` belongs to the same event as the path `raceId` and that both exist; it does not mutate race status.
