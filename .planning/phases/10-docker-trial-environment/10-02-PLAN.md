<objective>
Produce a demo-seed container that populates an empty (Flyway-migrated) database with the
"Wyvern RC Club" mid-season scenario, so the trial system is a realistic populated system
on first boot — not a blank slate (D-04, D-05).

Purpose: Wave 1 foundation — the trial compose stack (Plan 03) makes the `app` service
depend on `demo-seed` completing successfully before startup.
Output: `docker/seed/seed.sql` (idempotent) and `docker/seed/Dockerfile` (one-shot runner).

IMPORTANT — schema ordering: The seed runs AFTER Flyway migrations create the schema. The
compose dependency chain (Plan 03) is postgres(healthy) -> demo-seed(completes) ->
app(starts). But Flyway runs inside the `app` container at startup. This creates an ordering
problem: demo-seed needs the tables to exist, but the app (which creates them) waits for
demo-seed. RESOLUTION: the seed container must run Flyway-equivalent table creation OR wait
for tables. The simplest correct approach (used here): the seed SQL is written to be run by
`app` is wrong — instead, demo-seed depends only on postgres being healthy and the seed SQL
itself does NOT assume tables exist beyond a guarded check. See Task 1 action for the exact
mechanism: the seed SQL begins with a guard that exits cleanly if the `club_profiles` table
does not yet exist, and the compose `app` service depends on demo-seed with
`service_completed_successfully`; demo-seed is configured to retry. The DEFINITIVE mechanism
chosen here is documented in Task 1 — read it carefully.
</objective>

    Create `docker/seed/Dockerfile`. Build context for this image is `docker/seed/` (the
    compose `build:` will point at this directory — small, self-contained).

    Use `postgres:16-alpine` as the base because it bundles `psql` and `pg_isready`. The
    image copies `seed.sql` and runs it via `psql`. `psql` honours standard libpq env vars
    (`PGHOST`, `PGUSER`, `PGPASSWORD`, `PGDATABASE`) automatically — no connection string
    flags needed.

    ```dockerfile
    FROM postgres:16-alpine
    COPY seed.sql /seed.sql
    # libpq env vars (PGHOST/PGUSER/PGPASSWORD/PGDATABASE) supplied by compose.
    # -v ON_ERROR_STOP=1 makes psql exit non-zero if the seed SQL raises an exception,
    # so compose 'service_completed_successfully' correctly reflects seed success/failure.
    ENTRYPOINT ["sh", "-c", "psql -v ON_ERROR_STOP=1 -f /seed.sql"]
    ```

    The `-v ON_ERROR_STOP=1` flag is important: without it, psql exits 0 even after a SQL
    error, and compose would treat a failed seed as successful.
  </action>
