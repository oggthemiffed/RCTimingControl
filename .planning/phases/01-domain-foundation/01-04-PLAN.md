**JwtTokenService.java** — `@Service` in `security/` package. Uses JJWT 0.12.x API (NEVER 0.11 methods like `.setClaims()`, `.setExpiration()`):

```java
package dev.monkeypatch.rctiming.security;

@Service
public class JwtTokenService {
    private final SecretKey signingKey;
    private final long accessTokenTtlMs;
    private final long refreshTokenTtlMs;

    public JwtTokenService(
            @Value("${app.jwt.secret}") String base64Secret,
            @Value("${app.jwt.access-token-ttl-ms}") long accessTokenTtlMs,
            @Value("${app.jwt.refresh-token-ttl-ms}") long refreshTokenTtlMs) {
        this.signingKey = Keys.hmacShaKeyFor(Decoders.BASE64.decode(base64Secret));
        this.accessTokenTtlMs = accessTokenTtlMs;
        this.refreshTokenTtlMs = refreshTokenTtlMs;
    }

    public String generateAccessToken(User user) {
        return Jwts.builder()
            .subject(user.getId().toString())
            .claim("email", user.getEmail())
            .claim("firstName", user.getFirstName())
            .claim("lastName", user.getLastName())
            .claim("roles", user.getRoles().stream().map(Role::name).toList())
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + accessTokenTtlMs))
            .signWith(signingKey)
            .compact();
    }

    public String generateRefreshTokenValue() {
        // Random 32-byte token, hex-encoded
        byte[] bytes = new byte[32];
        new java.security.SecureRandom().nextBytes(bytes);
        return java.util.HexFormat.of().formatHex(bytes);
    }

    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(signingKey)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    public long getRefreshTokenTtlMs() { return refreshTokenTtlMs; }
}
```

Key points:
- Use `Jwts.builder().subject()` not `.setSubject()` (JJWT 0.12.x API)
- Use `Jwts.parser().verifyWith(key)` not `.setSigningKey()` (0.12.x)
- Roles stored as `List<String>` in JWT claims for stateless auth (per AUTH-04, AUTH-05)
- Refresh token is NOT a JWT — it's a random opaque value stored hashed in the database (per RESEARCH.md recommendation)

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http,
                                       JwtAuthenticationFilter jwtFilter) throws Exception {
    return http
        .csrf(AbstractHttpConfigurer::disable)
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/api/v1/auth/**").permitAll()
            .requestMatchers("/actuator/health").permitAll()
            .requestMatchers("/api/v1/admin/**").hasAnyRole("ADMIN", "RACE_DIRECTOR", "REFEREE")
            .anyRequest().authenticated()
        )
        .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
}

@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

- Session: STATELESS (no HttpSession created)
- `/api/v1/admin/**` requires any staff role (per AUTH-04)

**PasswordResetService.java** — `@Service @Transactional`:
- Constructor injection: `UserRepository`, `PasswordResetTokenRepository`, `RefreshTokenRepository`, `PasswordEncoder`, `JavaMailSender`
- `@Value("${app.password-reset.url:http://localhost:5173/reset-password}")` for the reset link base URL
- `requestReset(String email)`:
  1. Find user by email. If not found, return silently (NEVER reveal email existence — RESEARCH.md Pitfall 5, UI-SPEC 4.4)
  2. Generate random token (32 bytes, hex-encoded)
  3. Hash with SHA-256 before storing in `password_reset_tokens` table
  4. Set `expiresAt` to 1 hour from now
  5. Send email via `JavaMailSender` with link: `{baseUrl}?token={rawToken}`
- `confirmReset(String rawToken, String newPassword)`:
  1. Hash incoming token with SHA-256
  2. Find PasswordResetToken by hash
  3. If not found or `used=true` -> throw exception (400)
  4. If `expiresAt` is past -> throw exception (410 Gone)
  5. Update user's `passwordHash` with `passwordEncoder.encode(newPassword)`
  6. Mark token as `used = true`
  7. Revoke all refresh tokens for this user (`refreshTokenRepository.deleteByUser(user)`) — forces re-login on all devices

**AuthController.java** — `@RestController @RequestMapping("/api/v1/auth")`:

`POST /register`:
- Accept `@RequestBody @Valid RegisterRequest`
- Call `userService.createRacer(...)` — if email exists, DataIntegrityViolationException triggers 409 via GlobalExceptionHandler
- Generate access token, set refresh cookie, return 201 with AuthResponse

`POST /login`:
- Accept `@RequestBody @Valid LoginRequest`
- Find user by email, verify password with `passwordEncoder.matches()`
- If invalid: return 401 with generic message "Invalid email or password" (never specify which is wrong)
- If valid: generate access token, create RefreshToken in DB (store SHA-256 hash of raw token), set HttpOnly cookie, return AuthResponse

`POST /refresh`:
- Read `refresh_token` cookie from request
- Hash the cookie value with SHA-256, look up in `refresh_tokens` table
- If not found or revoked or expired: return 401
- Rotate: mark old token as revoked, create new RefreshToken, set new cookie, return new access token
- This implements token rotation for security

`POST /password-reset/request`:
- Accept `@RequestBody @Valid PasswordResetRequestDto`
- Call `passwordResetService.requestReset(email)`
- ALWAYS return `200 OK` — SECURITY REQUIREMENT (email enumeration prevention)

`POST /password-reset/confirm`:
- Accept `@RequestBody @Valid PasswordResetConfirmDto`
- Call `passwordResetService.confirmReset(token, newPassword)`
- On success: return 200
- On expired/used token: return 400 or 410

**Refresh cookie details:**
```java
ResponseCookie cookie = ResponseCookie.from("refresh_token", rawToken)
    .httpOnly(true)
    .secure(false)  // false for dev; true in production via config
    .sameSite("Lax")
    .path("/api/v1/auth/refresh")
    .maxAge(Duration.ofMillis(refreshTokenTtlMs))
    .build();
response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
```

The cookie path is `/api/v1/auth/refresh` — it is ONLY sent on refresh requests, not on every API call. This is correct because only the refresh endpoint needs the cookie.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → /api/v1/auth/* | Untrusted credentials and tokens from browser |
| JWT claims → SecurityContext | Token payload trusted after signature verification |
| Refresh cookie → database | Cookie value validated against hashed DB record |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Spoofing | JWT token | mitigate | JJWT `verifyWith(key)` rejects unsigned/tampered tokens; HMAC-SHA256 signing |
| T-01-10 | Spoofing | Login endpoint | mitigate | BCrypt password verification; generic "Invalid email or password" error prevents credential enumeration |
| T-01-11 | Information Disclosure | Password reset endpoint | mitigate | Always returns 200 OK regardless of email existence (prevents email enumeration per RESEARCH.md Pitfall 5) |
| T-01-12 | Elevation of Privilege | Role-based access | mitigate | `hasAnyRole("ADMIN", "RACE_DIRECTOR", "REFEREE")` on admin endpoints; roles extracted from JWT claims with "ROLE_" prefix for Spring Security |
| T-01-13 | Tampering | Refresh token reuse | mitigate | Token rotation: old token revoked on each refresh; database-backed tokens enable immediate revocation on password reset |
| T-01-14 | Information Disclosure | Token storage | mitigate | Refresh and reset tokens stored as SHA-256 hashes in database; raw values never persisted |
| T-01-15 | Elevation of Privilege | CSRF via cookie | mitigate | Access token sent via Bearer header (not cookie); refresh cookie restricted to `/api/v1/auth/refresh` path with `SameSite=Lax` |
| T-01-16 | Denial of Service | Brute-force login | accept | No rate limiting in Phase 1; documented as known gap in RESEARCH.md; acceptable for single-club deployment |
