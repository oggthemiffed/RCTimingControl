# UI Design Contract — Phase 1: Domain Foundation

**Phase:** 01-domain-foundation
**Written:** 2026-04-16
**Status:** Approved — ready for implementation

## 1. Scope

Phase 1 delivers the auth UI only. The React + Vite + Tailwind CSS + shadcn/ui scaffold is
created with routing in place for all future phases; the only screens built in Phase 1 are:

| Screen | Route | Auth Required |
|--------|-------|---------------|
| Login | `/login` | No |
| Register | `/register` | No |
| Password Reset Request | `/forgot-password` | No |
| Password Reset Confirm | `/reset-password?token=…` | No |

Admin configuration forms (club, track, format template CRUD) and all other screens are
**explicitly deferred to Phase 3 and beyond.** Stub routes are wired (`/admin/*`, `/racer/*`)
and render placeholder pages so routing infrastructure is exercised.

## 2. Theming & Design Tokens

### shadcn Preset

```
Preset ID: b1GyYWRfE
Init command: npx shadcn@latest init --preset b1GyYWRfE
```

Apply the preset exactly as shadcn generates it. The accent color must be **neutral blue**
(e.g. shadcn `blue` or `slate-blue` palette). If the preset `b1GyYWRfE` resolves to a
different accent, override `--primary` and `--ring` CSS variables in `@layer base` to a
neutral blue (e.g. HSL `221 83% 53%` for light, `217 91% 60%` for dark — standard shadcn
blue scale). Do not use orange, red, green, or other hues as the primary accent.

### Color Mode

Support both **light and dark mode**. Use the CSS variable approach that shadcn/ui emits by
default (`@layer base { :root { … } .dark { … } }`). Respect the OS preference via
`prefers-color-scheme` as the initial default (no manual toggle required in Phase 1 — add
one in Phase 2 or 3 when the nav shell is built).

Implementation: add `darkMode: "class"` in `tailwind.config.ts`. Apply the `.dark` class to
`<html>` based on `window.matchMedia('(prefers-color-scheme: dark)')` in a one-time effect
in `main.tsx`. This mirrors what shadcn documents for class-based dark mode.

### Typography

Use the font stack the preset defines. If the preset does not specify a font family, default
to the system-ui stack:

```css
font-family: ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont,
             "Segoe UI", Roboto, "Helvetica Neue", Arial, sans-serif;
```

## 4. Screen Specifications

### 4.1 Shared: `AuthLayout`

A centered single-column card layout for all auth screens. No navigation bar in Phase 1 (no
authenticated shell exists yet).

**Structure:**
```
<full-height flex-col flex-center bg-background>
  <div class="app-brand">   <!-- logo or text wordmark -->
    RC Timing Control
  </div>
  <Card class="w-full max-w-sm">
    <CardHeader>
      <CardTitle>{title}</CardTitle>
      <CardDescription>{subtitle}</CardDescription>
    </CardHeader>
    <CardContent>
      {form}
    </CardContent>
    <CardFooter>
      {secondary links}
    </CardFooter>
  </Card>
</full-height flex-col flex-center>
```

No sidebar. No topbar. Full viewport height, content vertically centered.

### 4.2 Login — `/login`

**Title:** Sign in  
**Subtitle:** Enter your email and password

**Fields:**

| Field | Type | Validation |
|-------|------|-----------|
| Email | `<Input type="email">` | Required, valid email |
| Password | `<Input type="password">` | Required |

**Actions:**
- Primary: `<Button type="submit">` — "Sign in" (full width)
- Secondary link in footer: "Don't have an account? Register" → `/register`
- Tertiary link below password field: "Forgot password?" → `/forgot-password`

**Behavior:**
- On submit: POST `/api/v1/auth/login` with `{ email, password }`
- On success: store access token in memory (`AuthProvider`), redirect to `/racer` (for
  `RACER` role) or `/admin` (for any staff role — `ADMIN`, `RACE_DIRECTOR`, `REFEREE`)
- On `401`: show field-level error "Invalid email or password" below the password field (do
  not reveal which credential is wrong)
- On network error: show toast "Unable to reach server. Please try again."
- Submit button shows a spinner and is disabled while the request is in flight

**Form library:** React Hook Form + Zod schema validation. Validate on `onBlur` for email
format; validate on submit for the full schema.

### 4.3 Register — `/register`

**Title:** Create account  
**Subtitle:** Join the club portal

**Fields:**

| Field | Type | Validation |
|-------|------|-----------|
| First name | `<Input type="text">` | Required, 1–100 chars |
| Last name | `<Input type="text">` | Required, 1–100 chars |
| Email | `<Input type="email">` | Required, valid email |
| Password | `<Input type="password">` | Required, min 8 chars |
| Confirm password | `<Input type="password">` | Required, must match password |

**Actions:**
- Primary: `<Button type="submit">` — "Create account" (full width)
- Footer link: "Already have an account? Sign in" → `/login`

**Behavior:**
- Client-side: validate matching passwords before submission; show inline error on confirm
  field if mismatch
- On submit: POST `/api/v1/auth/register` with `{ firstName, lastName, email, password }`
- On `201`: redirect to `/login` with a success toast "Account created. Please sign in."
- On `409` (email already registered): show field-level error on email field "An account
  with this email already exists"
- On `400` (validation failure): map field errors from the response body to the relevant
  form fields
- Submit button disabled and spinner shown while in flight

### 4.4 Forgot Password — `/forgot-password`

**Title:** Reset your password  
**Subtitle:** Enter your email and we'll send a reset link

**Fields:**

| Field | Type | Validation |
|-------|------|-----------|
| Email | `<Input type="email">` | Required, valid email |

**Actions:**
- Primary: `<Button type="submit">` — "Send reset link" (full width)
- Footer link: "Back to sign in" → `/login`

**Behavior:**
- On submit: POST `/api/v1/auth/password-reset/request` with `{ email }`
- **Always show success message regardless of whether the email exists** (prevents email
  enumeration): "If an account exists for that email, a reset link has been sent."
- Replace the form with the success message after submission (do not leave the form active)
- No error states beyond network failure (which shows a toast)

### 4.5 Reset Password Confirm — `/reset-password`

Accessed via the link in the password reset email. The `token` query parameter is extracted
from the URL.

**Title:** Set new password  
**Subtitle:** Choose a strong password for your account

**Fields:**

| Field | Type | Validation |
|-------|------|-----------|
| New password | `<Input type="password">` | Required, min 8 chars |
| Confirm new password | `<Input type="password">` | Required, must match |

**Actions:**
- Primary: `<Button type="submit">` — "Set password" (full width)

**Behavior:**
- On mount: if `token` query param is missing, redirect immediately to `/forgot-password`
  with toast "Invalid or expired reset link. Please request a new one."
- On submit: POST `/api/v1/auth/password-reset/confirm` with `{ token, newPassword }`
- On `200`: redirect to `/login` with success toast "Password updated. Please sign in."
- On `400`/`410` (token expired or already used): show error card "This reset link has
  expired or has already been used. Please request a new one." with link to `/forgot-password`
- Submit button disabled and spinner shown while in flight

## 5. Routing

Use React Router v6 Data Router (`createBrowserRouter`).

```
/                   → redirect to /login (or /racer if authenticated)
/login              → LoginPage (public)
/register           → RegisterPage (public)
/forgot-password    → ForgotPasswordPage (public)
/reset-password     → ResetPasswordPage (public, token from query param)
/racer/*            → RacerPlaceholderPage (protected — RACER or any staff role)
/admin/*            → AdminPlaceholderPage (protected — ADMIN, RACE_DIRECTOR, or REFEREE)
*                   → 404 page (simple "Page not found" with link back to /login)
```

**Route protection:** A `<ProtectedRoute>` wrapper component checks the auth context. If
no valid token, redirect to `/login` preserving the `from` location so the user is returned
after login.

**Role-based redirect on login:**
- If token contains role `ADMIN`, `RACE_DIRECTOR`, or `REFEREE` → navigate to `/admin`
- Otherwise (role `RACER`) → navigate to `/racer`

## 6. Auth State Management

### Token Strategy

| Token | Storage | Lifetime |
|-------|---------|---------|
| Access token (JWT) | In-memory React state (`AuthProvider`) | 15 minutes (backend-configured) |
| Refresh token | `HttpOnly` cookie (set by backend) | 7 days (backend-configured) |

**Do not store the access token in `localStorage` or `sessionStorage`.** In-memory storage
means the token is lost on page refresh; a background refresh request on mount (using the
`HttpOnly` cookie) silently restores the session if the refresh token is valid.

### `AuthProvider` contract

```typescript
interface AuthContextValue {
  user: AuthUser | null;          // null = unauthenticated
  accessToken: string | null;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
  isLoading: boolean;             // true during the initial session restore check
}

interface AuthUser {
  id: string;
  email: string;
  firstName: string;
  lastName: string;
  roles: Array<'RACER' | 'ADMIN' | 'RACE_DIRECTOR' | 'REFEREE'>;
}
```

On mount, `AuthProvider` attempts `POST /api/v1/auth/refresh` silently. If it succeeds, user
and access token are populated. If it fails (no cookie, or expired), user remains `null`.
While this check is in flight, `isLoading = true` — render a centered `<Spinner>` rather
than the router so unauthenticated redirects don't flash before the check completes.

### `api.ts` — Base HTTP client

Configure an Axios instance (or `fetch` wrapper) that:
1. Attaches the in-memory access token as `Authorization: Bearer {token}` on every request
2. On `401` response: attempts a token refresh, then retries the original request once
3. On second `401` after refresh: calls `logout()` and redirects to `/login`

## 7. Error Handling & Feedback Patterns

These patterns apply uniformly across all Phase 1 screens and must be established as
conventions for all future phases.

### Field-level errors

Use React Hook Form's `formState.errors` to render error messages directly below each
field. Use the shadcn `<FormMessage>` component (or equivalent pattern from the preset).
Error text is `text-destructive text-sm`.

### Toast notifications

Use shadcn's `<Toaster>` + `useToast()` hook (or the `sonner` alternative if the preset
pulls it in). Mount `<Toaster>` once in `App.tsx`. Toasts are used for:
- Success confirmations that follow a page navigation (account created, password updated)
- Network/server errors that cannot be mapped to a specific field

Toast duration: 5 seconds for success, 8 seconds for errors (long enough to read).

### Loading states

Disable the submit button and replace its label with a spinner while a mutation is in
flight. Use a `<Loader2>` icon from `lucide-react` with `animate-spin`. Do not show a
full-page loading overlay for form submissions.

### 404

A minimal page: heading "Page not found", short message, and a link back to `/login`. No
auth layout wrapper needed — plain centered text is sufficient.

## 8. Accessibility Requirements

- All form fields must have an associated `<label>` (shadcn `<FormLabel>` handles this)
- Error messages must be associated with their field via `aria-describedby`
- Submit buttons must not be `aria-disabled` — use the `disabled` attribute so screen
  readers announce the state correctly
- Color contrast must meet WCAG AA in both light and dark mode (the shadcn preset's default
  palette satisfies this for standard components)
- Focus ring must be visible — do not suppress the browser's default focus outline on
  interactive elements (shadcn preserves this with `focus-visible` utilities)
- The auth layout card must not trap keyboard focus; Tab order follows DOM order

## 9. Dev Tooling

### Vite dev server proxy

Configure the Vite dev server to proxy `/api` to the Spring Boot backend:

```typescript
// vite.config.ts
server: {
  proxy: {
    '/api': 'http://localhost:8080'
  }
}
```

This avoids CORS issues in development without configuring Spring's CORS allowlist for the
dev origin.

### Environment variables

```
VITE_API_BASE_URL=    # empty in dev (proxy handles it); set to backend origin in production
```

The `api.ts` base URL defaults to `''` (relative, uses proxy) when the env var is unset.

- [ ] All form fields have visible labels and accessible error messages

- [ ] Vite proxy routes `/api` calls to `localhost:8080` in dev without CORS errors
- [ ] TypeScript strict mode: `tsc --noEmit` passes with zero errors
